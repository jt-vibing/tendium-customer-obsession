Agent-First Vision

Customer Graph V2

The Customer Graph isn't an app you open. It's an agent that works alongside you. One intelligence layer that knows every customer — powered by data no competitor can access.

The Paradigm Shift
V1 Thinking
Build a better dashboard
Search pages, profile tabs, glossary, punch cards. The tool shows data. The human navigates, reads, synthesizes, and decides.
→
Agent Thinking
Make work disappear
One agent that already knows the customer. It briefs you, alerts you, and recommends what to do next. You don't search for intelligence — it finds you.
The question isn't
"What features should this app have?" — it's "What work should disappear entirely?"

Agents vs Sub-Agents vs Skills
Concept Analogy Autonomy Tendium Example
Agent A team member with a job Full — decides HOW to achieve a goal “Prepare me for my Stockholm call” → agent decides what to look up, what's relevant, how to present it
Sub-agent A specialist the agent delegates to Narrow — executes a specific reasoning task Revenue Analyst: “Calculate NRR and expansion signals for this account”
Data Skill A function that fetches data None — input/output only get_arr(org_nr) → returns 245,000 SEK
Action Skill A function that does something None — input/output only send_slack_alert(channel, msg) → posts churn risk to #cs-alerts
Architecture — 4 Layers
Layer 4

Personas (Interface Modes)

Chat interface, scheduled briefs, Slack alerts, CLI. Different entry points — same agent underneath.

Layer 3

The Agent (Orchestrator)

One Customer Graph Agent. Receives questions, determines which skills to call, delegates to sub-agents, synthesizes across all sources, generates actionable briefs.

Layer 2

Sub-Agents (Specialized Reasoning)

Revenue Analyst • Engagement Analyst • Relationship Intelligence • Risk & Opportunity Detector

Layer 1

Skills (Data + Action)

Data: Younium • PostHog • Garba • Procurement • HubSpot • Company Groups
Action: Analyze & Visualize • Write Brief • Notify • Calendar • Search • Memory

Build from bottom up: Skills first, then Sub-agents, then Agent, then Personas.

V1 Features Become Agent Skills
The insight
V1's "features" are actually the agent's skills in disguise. Nothing is wasted — everything transforms.
V1 Feature Becomes Agent Skill How
Search page search_customers(query) Agent calls this to find companies
Profile page get_customer_profile(org_nr) Agent calls this to pull full context
Glossary System prompt Shared definitions baked into how it reasons
Punch cards Agent output format Generated on demand, not pre-rendered
Data connectors MCP tools Agent invokes Younium, PostHog, Garba directly
Two Mental Models
V1 Ship
Model A: App + Agent
The agent is a feature inside the app
App (Search, Profile, Glossary) ├── Chat panel (agent sidebar) └── Structured views remain primary
Like adding Copilot to an existing IDE. The app is the product. The agent enhances it. Safe bet. Tangible for stakeholders.
Vision
Model B: Agent + Surfaces
The app is just one output format
Agent (the product) ├── "How's the portfolio?" → dashboard ├── "Brief me on Stockholm" → brief ├── Monday 8am → Slack digest └── Health drop → Slack alert
The agent IS the product. Web, Slack, email are surfaces it renders to. Intelligence takes the shape you need.
Decision
Ship V1 as Model A — structured app with agent chat panel. Proves skills work. Gives stakeholders something tangible.

Evolve toward Model B — as the agent proves itself, structured pages become less important. The agent starts rendering its own views. The "app" becomes just one surface.

The Demo — A Day in the Life

This isn't a chat demo. This is Monday at Tendium — where the agent works across every surface, proactively, before anyone asks.

07:55 Automatic
Agent wakes up. No one asked it to.
It pulls overnight data from Younium, PostHog, and Garba. Detects that Adlibris usage dropped 40% last week while their renewal is in 45 days. Flags it as a churn risk.
Sends Slack DM to CS owner: "Adlibris health alert — usage down 40%, renewal in 45 days. Here's what I'd do." Generates a 3-point intervention playbook with evidence from 4 sources
08:00 Slack
Hugo's Monday report arrives. He didn't build it.
#commercial-ops gets a formatted digest: 42.1M ARR (+180K this week), NRR 116%, 3 at-risk accounts, 7 expansion signals. Each company name is a deep link. Hugo reads it in 30 seconds instead of building it in 5 hours.
Renders: interactive portfolio dashboard card with sparklines Sources cited: Younium (ARR), PostHog (usage), Garba (recent calls), Procurement (pipeline)
09:15 Calendar trigger
Sales AE has a call with Stockholm Municipality at 10:00. Agent sees it in the calendar.
45 minutes before the call, the agent sends a structured brief to the AE's Slack DM. Not a wall of text — a visual card:
Stockholm Municipality Scale plan • 385K SEK ARR
Health Strong
Renewal 82 days
Last meeting Jan 28
Open tenders 3 active
Talking points: They mentioned budget review in last Garba call. 3 new tenders match their profile — use as expansion hook. Their team added 4 new users last month (PostHog).
Watch out: Competitor Mercell won a co-bid on their latest procurement.
11:30 Chat
Product lead opens the agent and asks a question no dashboard could answer.
"Which Scale customers increased usage this quarter but haven't adopted BidFlow? Give me names, usage delta, and what they're actually doing instead."
The agent queries PostHog (usage trends), Younium (plan tier), and Garba (recent conversations). Returns a table of 12 companies with usage deltas, feature maps, and quotes from CS calls about why they haven't adopted BidFlow. This insight would take a human 2 days to compile. The agent does it in 8 seconds.
16:00 Proactive
New procurement data drops. Agent spots something no one was looking for.
A batch of 230 new tender awards loaded into the procurement database. Agent cross-references against Company Groups and finds that 3 non-customers in the Adlibris group just won tenders worth 12M SEK combined. These are ultra-warm leads — same parent company as an existing customer.
Sends expansion signal to Sales with company profiles, group relationship, and tender details Drafts a warm outreach message: "Your colleagues at Adlibris use us for..."
The point
Nobody opened an app. Nobody searched. Nobody built a spreadsheet.
The agent worked across 5 surfaces (Slack DM, channel post, visual card, chat response, proactive alert), pulled from 6 data sources, and generated 4 different output formats — all before anyone had to ask the right question.

Principle

4 Root Causes

Root Cause 1: No aggregate query layer
29 questions (58%). Skills today query one company at a time. Need 5 new aggregate skills.
Root Cause 2: No temporal data
8 questions (16%). No start_date, churned_date, or event log. Point-in-time snapshot only.
Root Cause 3: No cross-portfolio text
3 questions (6%). Can't search/aggregate across all meeting summaries or MEDDIC insights.
Root Cause 4: No external data
5 questions (10%). Need real procurement DB, HubSpot, Allabolag TAM. V2 integrations.
Build Plan — Skills First, Cards Second
Order of operations
The temptation is to jump to building pretty cards. But cards without data are empty shells. Aggregate skills are invisible to the user but they're the foundation — the agent can't answer 29 of 50 questions without them.

Skills first. Cards second. Enhancements third.
